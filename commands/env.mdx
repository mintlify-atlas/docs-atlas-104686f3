## Overview

The `env` command helps you manage multiple Typesense environments (development, staging, production) using `.env` files. It provides subcommands to list available environments and show current configuration.

## Subcommands

- [`list`](#list) - List available environments
- [`show`](#show) - Show current environment configuration

## Environment Files

tsctl uses environment files to manage connection settings:

- `.env` - Default/base environment
- `.env.development` - Development environment
- `.env.staging` - Staging environment  
- `.env.production` - Production environment
- `.env.<custom>` - Any custom environment name

Generate these files with:
```bash
tsctl init --with-environments
```

## Switching Environments

Use the global `--env` flag on any tsctl command:

```bash
tsctl plan --env production
tsctl apply --env staging
tsctl drift --env development
```

## list

List all available environment files in the current directory.

### Syntax

```bash
tsctl env list
```

### Examples

#### List environments

```bash
tsctl env list
```

**Output:**
```
Available environments:

  ● (default) (.env)
  ○ development (.env.development)
  ○ staging (.env.staging)
  ○ production (.env.production)

  Use --env flag to switch:
    tsctl plan --env production
    tsctl apply --env staging -y
```

#### No environment files

```bash
tsctl env list
```

**Output:**
```
Available environments:

  No environment files found.

  Create .env files with: tsctl init --with-environments
```

### Use Cases

- **Discover environments** - See what environments are configured
- **Verify setup** - Check that environment files exist before deployment
- **Documentation** - Show team members available environments

## show

Show the current environment's Typesense connection configuration and test the connection.

### Syntax

```bash
tsctl env show [--env <name>]
```

### Examples

#### Show default environment

```bash
tsctl env show
```

**Output:**
```
Environment: (default)

  Host:     localhost
  Port:     8108
  Protocol: http
  API Key:  ●●●●●●●● (set)
  URL:      http://localhost:8108

✓ Connected to Typesense
```

#### Show production environment

```bash
tsctl env show --env production
```

**Output:**
```

  Port:     443
  Protocol: https
  API Key:  ●●●●●●●● (set)
```

## Environment Variables

Each environment file should define:

```bash
# .env.production
TYPESENSE_HOST=typesense.example.com
TYPESENSE_PORT=443
TYPESENSE_PROTOCOL=https
TYPESENSE_API_KEY=your-production-api-key
```

### Variable Precedence

1. **Base .env** is loaded first
2. **Environment-specific .env.{'{'}name{'}'}** overrides base values

Example:
```bash
# .env (base)
TYPESENSE_HOST=localhost
TYPESENSE_PORT=8108
TYPESENSE_API_KEY=dev-key

# .env.production (overrides)
TYPESENSE_HOST=prod.typesense.example.com
TYPESENSE_PORT=443
TYPESENSE_PROTOCOL=https
TYPESENSE_API_KEY=prod-key
```

When you run `tsctl plan --env production`:
- Loads `.env` first
- Then loads `.env.production` (overrides values)
- Results in production settings

## Multi-Environment Workflow

### Setup

```bash
# Initialize with environment files
tsctl init --with-environments

# Update each .env file with correct credentials
vim .env.development
vim .env.staging  
vim .env.production

# Verify each environment
tsctl env show --env development
tsctl env show --env staging
tsctl env show --env production
```

### Development to Production

```bash
# 1. Develop locally
tsctl plan --env development
tsctl apply --env development

# 2. Test in staging
tsctl plan --env staging
tsctl apply --env staging

# 3. Deploy to production
tsctl plan --env production
tsctl apply --env production
```

### Environment Isolation

Each environment has:
- **Separate Typesense instance** (different host/credentials)
- **Separate state** (stored in each instance's `tsctl_state` collection)
- **Independent resources** (collections, aliases, etc.)

Changes in one environment don't affect others.

## Security Best Practices

<Warning>
  **Protect your environment files:**
  
  1. **Add to .gitignore** - Never commit `.env` files with real credentials
  2. **Use secrets management** - Store production API keys in vault/secrets manager
  3. **Separate credentials** - Use different API keys per environment
  4. **Limit permissions** - Use read-only keys where possible
  5. **Rotate regularly** - Change API keys periodically
</Warning>

### Example .gitignore

```bash
# .gitignore
.env
.env.*
!.env.example
```

### Example .env.example

```bash
# .env.example (safe to commit)
TYPESENSE_HOST=localhost

TYPESENSE_API_KEY=your-api-key-here
```
